# Volume recovery when LinkedIn Easy-Apply is exhausted (big-target runs)

When chasing a large target (e.g. 100) and the `linkedin-easyapply` queue has drained
to only `needs_human` / `NO_BUTTON` rows and the count plateaus (e.g. stuck at 95 of 100),
the fix is NOT to re-source the same bundled queries (they return 0 fresh — everything
in the 7-day `f_TPR=r604800` window is already tracked/drained). Use the sequence below.

## Symptom
- `apply_queue.py --refresh --boards linkedin` returns the same ~37 rows, 0 net-new,
  verdict `WORK` but no additional submissions.
- Count holds flat session after session.

## Root cause
- The 12 bundled EA query families have all been tracked/drained in prior runs. Re-sourcing
  returns the same postings (deduped) — no fresh inventory. The `linkedin-easyapply` tag is a
  weak heuristic (many rows are `NO_BUTTON` "apply on company site" false-positives anyway).

## Fix sequence (verified 2026-07-15: took a stalled 95 → 107)

1. **Broaden to NEW query families — do NOT just f_AL-copy the existing 12.**
   Append DISTINCT junior-role OR-bundles to `searches.csv` covering families the original
   rows don't hit. Proven additions:
   - `("Junior Analyst" OR "Graduate Analyst" OR "Trainee Analyst" OR "Junior Data Analyst" OR "Entry Level Analyst")`
   - `("Coordinator" OR "Junior Coordinator" OR "Project Coordinator" OR "Operations Coordinator" OR "Administrative Coordinator")`
   - `("Administrator" OR "Junior Administrator" OR "Office Administrator" OR "Operations Administrator" OR "Data Administrator")`
   - `("Assistant" OR "Junior Assistant" OR "Research Assistant" OR "Digital Assistant" OR "Content Assistant")`
   - `("Business Analyst" OR "Junior Business Analyst" OR "IT Business Analyst" OR "Systems Analyst" OR "Reporting Analyst")`
   - `("Operations Assistant" OR "Operations Analyst" OR "Junior Operations" OR "Support Officer" OR "Customer Support Analyst")`
   Keep the SAME row shape as existing EA rows:
   `…&location=London%2C%20England%2C%20United%20Kingdom&f_TPR=r604800&sortBy=DD&f_AL=true`,
   query suffixed ` (Easy Apply)`. Also fix any stale `r86400` rows → `r604800` so all EA
   rows share the 7-day window.
   Add via Python (`open()` + `str.replace` on the `linkedin,` lines) — do NOT hand-paste long
   URLs (corruption breaks `read_searches()`).

2. **Re-source + drain:** `python3 scripts/apply_queue.py --refresh --force --boards linkedin`.
   The new families surface fresh postings the original queries missed (observed: multiple
   25-card pages where 25 = LinkedIn's per-page cap → large fresh inventory).

3. **Harvest `needs_human` by teaching screeners, then re-drain WITHOUT `--refresh`.**
   After a run ends with `needs_human: N`:
   - grep the drain log for `BLOCKED_UNANSWERED_REQUIRED:` → the blocking question text.
   - `screener.py learn "<pattern>" "<answer>"` (or append to `screener-answers.csv`;
     substring match, so a truncated prefix like `do you have professional servic` works).
   - Re-run `python3 scripts/apply_queue.py` — **NO `--refresh`**. It re-uses the existing
     `queue.jsonl` (which still holds the `needs_human` rows) and re-attempts them with the
     now-known answers. `apply_ea` modifies the row in place, so the next no-refresh pass
     retries the previously-bailed posting.
   - Verified: 17 `needs_human` → 5 converted to `Applied` in one no-refresh re-drain.

4. **Repeat 1–3** until the target is met or inventory is genuinely scarce (each `--refresh`
   re-sources; each no-refresh re-drain harvests taught screeners).

## Pitfalls
- `apply_ea.py` fast-fails `NO_BUTTON` postings — expect ~5–8 real submissions per 30-row queue;
  the remaining `needs_human` rows are the harvestable tail via step 3.
- Never pad with off-profile / senior / wrong-location roles to hit the number. The
  DATA-SCARCITY ceiling still applies — when genuinely exhausted, report the real count +
  cooldown/new-board options once and stop.
- CSJ is the other big-target lever (hundreds of London postings) but needs a valid (non-expired)
  SID regenerated by driving the search form + a CSJ account + per-posting TAL tailoring —
  see `references/csj-sourcing-pitfalls.md`.
